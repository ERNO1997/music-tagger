## Context

`path` (`files.path TEXT PRIMARY KEY` in `internal/infrastructure/persistence/sqlite_store.go`) has been an immutable identifier since the very first change: `TrackingStore.LoadAll` returns records keyed by it, and every other port method (`Get`, `RecordIdentification`, `RecordCoverArt`, `GetCoverArtPath`, `RecordLyrics`, `GetLyrics`, `RecordTagged`) takes `path` as its lookup key (`internal/usecases/ports.go`). Nothing before this change has ever needed to change a file's location — scan discovers it, identify/enrich/tag all mutate fields on the row at that path. Relocation is the first capability whose entire job is to change where the file physically is, which means it's also the first to need to change the value of that identifier.

The existing background-job pattern (`JobManager` → `TagManager`/`EnrichManager`/`IdentifyManager` → `*_handler.go`) already solves "run this over N selected paths, in the background, one job at a time, with pollable progress" four times over (scan, identify, enrich, tag). Relocation reuses that pattern as a fifth, independent job type rather than inventing a new one.

## Goals / Non-Goals

**Goals:**
- Physically move an already-identified-and-tagged file into `/music/{sanitized artist}/{sanitized album}/{zero-padded track} - {sanitized title}.ext`, on demand, per selection.
- Keep the move atomic per file: either it fully succeeds (directory created, file moved, tracking record's path updated to match) or the source file is left exactly where it was.
- Update the tracking store so every other capability (identify, enrich, tag, the API, the UI) keeps working against the file's new path with no special-casing.

**Non-Goals:**
- Retagging or re-verifying tags as part of relocation — tagging is a prerequisite (gated on), not something this change re-runs.
- Automatic relocation as part of scan, identify, enrich, or tag (stays on-demand/human-triggered, same as all four).
- Deduplicating or merging files that already exist at a computed destination — that's a collision, and this change treats it as a per-file failure (see Decisions), not something to resolve automatically.
- General-purpose distributed locking across job types — scan and relocate get a specific, narrow mutual-exclusion guard (see Decisions) because they're the one pair that can corrupt tracking state if interleaved; identify/enrich/tag remain independent of everything, as before.

## Decisions

### Introduce a surrogate `id INTEGER PRIMARY KEY`; `path` becomes a mutable, uniquely-indexed column
Originally this design kept `path TEXT PRIMARY KEY` and updated it in place via a plain `UPDATE`, reasoning that SQLite permits changing a `TEXT PRIMARY KEY` value directly and this schema has no foreign keys to cascade. On reflection, a surrogate key is worth the (small) extra cost: it makes "this is the same row; `path` is just an attribute of it" the literal schema truth rather than something a reader has to reason through, and it costs almost nothing at the Go level — **every existing `TrackingStore` port method (`Get`, `RecordIdentification`, `RecordCoverArt`, `GetCoverArtPath`, `RecordLyrics`, `GetLyrics`, `RecordTagged`) keeps its exact same signature and `WHERE path = ?` SQL**, since `path` remains a `UNIQUE NOT NULL` indexed column. `RecordRelocation`'s implementation is unchanged too: `UPDATE files SET path = ?, relocated = 1, relocate_error = '', updated_at = ? WHERE path = ?` still identifies the row by its old `path` value.

The real cost is entirely in schema migration, not application code (see Migration Plan): SQLite cannot alter an existing table's declared `PRIMARY KEY` in place the way `ALTER TABLE ADD COLUMN` handles every column addition so far — moving an already-deployed database from `path TEXT PRIMARY KEY` to `id INTEGER PRIMARY KEY` requires a one-time full table rebuild.

### Relocation requires `status == identified AND Tagged == true`
Mirrors the existing skip-and-log pattern (`EnrichManager.Start`, `TagFile.Tag`) but with a stricter gate: a file that's `identified` but not yet `tagged` is skipped (logged, not an error), not relocated. Rationale: the charter's Phase C lists Enrich → Tag → Relocate in that order, and relocation is the most disruptive, least reversible step so far (it changes the file's actual location in the user's real folder tree, not just an in-place edit) — gating it on `Tagged` means a file is never moved into a folder named after metadata that isn't also already embedded in the file's own tags, keeping "where it lives" and "what it says about itself" in sync at the one point where they could otherwise silently diverge.

### `RelocateFile` reads from the tracking store itself, same as `TagFile`
`RelocateFile.Relocate(ctx, path)` loads the file's `FileRecord` via the existing `TrackingStore.Get` (added for tagging), builds sanitized path segments from the resolved metadata already on that record, and calls the `Relocator` port — no new DTO-assembly burden on `RelocateManager`, consistent with how `TagFile` already works.

### Path sanitization and destination construction live in `internal/infrastructure/filestat/relocator.go`
A `sanitizeSegment(s string) string` strips `\ / : * ? " < > |` (charter §4.3's exact list) from a path segment. Track numbers are formatted `%02d`. The destination is `filepath.Join(musicRoot, sanitizeSegment(artist), albumDirName(year, album), fmt.Sprintf("%02d - %s%s", track, sanitizeSegment(title), ext))`. Sanitization runs before both `os.MkdirAll` and `os.Rename`, never after (charter §4.3's explicit ordering requirement) — there is exactly one code path that builds the destination string, so this ordering can't be gotten wrong by a caller forgetting a step.

### Album directory is prefixed with release year, when known
`albumDirName(year, album)` renders `"{year} - {sanitized album}"` when `year > 0`, or just `"{sanitized album}"` when the release has no usable date (`year <= 0`) — omitted rather than shown as a placeholder like `"Unknown - Album"`, consistent with how every other optional resolved field in this app (release year in the API response, track/disc totals, cover art, lyrics) is already just left out when unavailable rather than faked. `RelocateInput` gained a `Year int` field (mirroring `RecordingMetadata.Year`/`FileRecord.Year`, already 0 when unresolved) so `RelocateFile` can pass it through from the tracked record with no new lookup.

### Destination collision is a per-file failure, not an auto-rename or overwrite
If the computed destination path already exists on disk, `Relocator.Relocate` returns an error rather than overwriting it or picking an alternate name (e.g. appending "(2)"). Silent overwriting risks destroying a different, unrelated file that happens to already live at that canonical path (e.g. two different recordings that both resolved to `Artist/Album/01 - Title.mp3` due to a MusicBrainz data quirk); silent auto-renaming hides a real data problem (most likely: this file's identification is wrong, or a genuine duplicate exists) behind a name the user didn't ask for. Consistent with "atomic per-file... on any error the source file is left untouched" (proposal.md).

**Found via live testing (task 8.4)**: this collision check as originally written also fired when a file's computed destination was *itself* — e.g. relocating an already-relocated file again after a re-identification that resolved back to the same identity (`os.Stat(dest)` succeeds because `dest` and the file's current `path` are the same location, so it looks exactly like a collision even though nothing needs to move). This makes a very ordinary action — re-running relocate over a selection that includes already-correctly-placed files — fail with a misleading "destination already exists" error. Fixed by checking `filepath.Clean(dest) == filepath.Clean(path)` first: if the file is already exactly where it should be, `Relocate` returns success immediately as a no-op, without touching the filesystem at all. Re-verified afterward that a *genuine* collision — a different tracked file whose resolved identity happens to compute to the same destination as another file already there — still correctly fails and leaves both files untouched.

### Empty source directories are removed after a successful move, up to (not including) the music root
After `os.Rename` succeeds, `Relocate` walks upward from the file's original directory, removing each directory that's now empty, stopping at the first directory that still contains something real or at `musicRoot` itself (never above it — a `strings.HasPrefix` check against the cleaned root guards this). `Undo`'s rollback does the same walk from the directory the file was briefly relocated into, for the same reason: an aborted relocation shouldn't leave litter either.

"Empty" disregards OS-generated junk files — notably macOS's `.DS_Store`, which Finder recreates in nearly every directory it has ever browsed (confirmed appearing in this project's own test output), so a strict zero-entries check would rarely fire in practice. A directory containing only recognized junk (`.DS_Store`, `Thumbs.db`, `desktop.ini`, AppleDouble `._*` sidecar files) has that junk deleted along with the directory; a directory containing any real file or subdirectory is left untouched. This cleanup is best-effort and silent on failure (a permissions issue, a directory that turns out not to be empty) — it's cleanup, not part of the relocation's correctness, so a failure here doesn't fail the relocation that triggered it.

### A failed database write after a successful move triggers a best-effort rollback
The "atomic per-file" promise (source untouched on any error) as originally scoped only covered `Relocator.Relocate`'s own `MkdirAll`+`Rename` step — it didn't extend to the follow-up `TrackingStore.RecordRelocation` call. If the physical move succeeds but that database write fails (disk error, unexpected crash, etc.), the file would sit at its new location while the tracking store still says the old one — and a later scan would see the old path as `missing` (history preserved, per existing behavior) while discovering the new path as a brand-new, unidentified file, since re-scan only merges "missing → reappeared" by matching the *same* path, not by fingerprint. The old row's resolved identity would be effectively orphaned.

Fix: `Relocator` gets one more method, `Undo(ctx context.Context, currentPath, originalPath string) error` — a bare `os.Rename(currentPath, originalPath)`, no sanitization or `MkdirAll` needed since the original directory already existed. `RelocateFile.Relocate` calls `Undo` if `RecordRelocation` fails, moving the file back before returning the error, so a failure at this step keeps the same "nothing changed" guarantee the filesystem step already has. If `Undo` itself also fails (the genuinely rare double-failure case), the error message states plainly that the file is now physically at the new path but tracked at the old one, and it's logged via the manager's existing `log.Printf` skip/error path so an operator watching server logs can see it — this last, narrow case is accepted as a visible failure rather than something further code can reliably self-heal (at that point two independent operations have both failed; a third automated action isn't obviously safer than a clear error).

### Re-identification clears `relocated`/`relocate_error`, same as it clears `tagged`/`tag_error`
`SQLiteStore.RecordIdentification` already resets `cover_art_path`, `lyrics`, `synced_lyrics`, `tagged`, `tag_error` to blank in both its identified and not-identified branches, since a re-identification can resolve to a different recording than before. This change adds `relocated = 0, relocate_error = ''` to that same reset — a file's prior relocation (or relocation failure) was computed against a possibly-different prior identity and isn't guaranteed to still be correct. Note this does **not** move the file back — the tracking record's path already reflects wherever the file physically is; only the *outcome flag* is cleared, prompting the user to re-run relocation (which will then compute a fresh destination from the new identity) if needed.

### Endpoint and job shape: identical to tag, except scan and relocate now mutually exclude each other
`POST /api/v1/library/relocate` (body: `{"paths": [...]}`) → `202 Accepted`, backed by a new `RelocateManager` (its own `JobManager`). `GET /api/v1/library/relocate/status` for polling. Skip-and-log for any path that isn't `identified && Tagged`, exactly like the existing managers' skip conditions. Relocate remains independent of identify/enrich/tag (no shared guard, as before) — **only its relationship with scan changes**, per the next decision.

### Scan and relocate mutually exclude each other; a small `StatusChecker` closes the loop without a construction-order dependency
Originally this change accepted the scan/relocate race as an unmitigated risk (a scan walking `/music` while a relocate job moves a file could briefly see it as both "missing" at the old path and "new" at the new one). On reflection, a real fix is small enough to be worth doing rather than documenting around: both directions need to reject a start attempt while the other's job is running, not just one — blocking relocate-while-scanning alone still leaves scan-while-relocating as a live version of the exact same race.

Implementation shape: a package-level interface,
```go
type StatusChecker interface {
    Status() JobStatus
}
```
which every existing manager (`RefreshManager`, `IdentifyManager`, `EnrichManager`, `TagManager`, the new `RelocateManager`) already satisfies via its existing `Status()` method — no new type needed on their side. `RelocateManager` takes a `StatusChecker` (the `RefreshManager`, which already exists earlier in `main.go`'s composition order) at construction and checks `.Status().Running` before starting. The reverse direction has a construction-order wrinkle: `RefreshManager` is built long before `RelocateManager` exists in `main.go`. Rather than reordering the whole composition root, `RefreshManager` gets one new method, `SetRelocateStatus(s StatusChecker)`, called once `RelocateManager` exists (still before the automatic startup scan is triggered) — `RefreshManager.Start` then checks it (nil-safe: does nothing if never set).

This has the same check-then-act race window any of this codebase's existing job guards have (a `Status()` read followed by a separate `Start()` isn't atomic against a concurrent status change) — accepted as consistent with the existing guard style (each `JobManager` already only mutex-protects its own start/status, not cross-manager coordination) and not worth a heavier cross-manager lock for a human clicking two buttons within the same instant.

### Cross-manager blocking needs its own sentinel errors — the existing ones are all secretly the same value
**Found via live testing** (task 8.2 verification): the plan above originally called for `RelocateManager.Start` to return the existing `ErrRefreshInProgress` when blocked by a scan, and for `RefreshManager.Start` to return the existing `ErrRelocateInProgress` when blocked by a relocate job — reasoning that reusing already-exported names was simpler than adding new ones. This doesn't actually work: `ErrRefreshInProgress`, `ErrIdentifyInProgress`, `ErrEnrichInProgress`, `ErrTagInProgress`, and `ErrRelocateInProgress` are **all literally the same value** (each is declared as `= ErrJobInProgress`, not a new error wrapping it), because until this change no single `Start()` call ever needed to distinguish two different causes of "something is already running." Once `RelocateManager.Start` had two possible causes (its own job busy, or scan busy) and both were expressed as values equal to `ErrJobInProgress`, `errors.Is` could not tell them apart — the handler's first `errors.Is` check always matched, regardless of the real cause. Confirmed live: blocking a relocate request during an active scan returned "a relocate job is already in progress" instead of the correct "a scan refresh is in progress."

Fix: two new, genuinely distinct sentinels — `ErrBlockedByScan` (in `relocate_manager.go`) and `ErrBlockedByRelocate` (in `refresh_manager.go`), each its own `errors.New(...)`, not aliased to `ErrJobInProgress` or to each other. `RelocateManager.Start` now returns `ErrBlockedByScan`; `RefreshManager.Start` now returns `ErrBlockedByRelocate`. The handlers check the correct pair of sentinels per direction. Re-verified after the fix: both directions now report the correct, distinguishable reason.

### Tagging invalidates its own scan baseline — found via live testing, fixed as part of this change
**Found via live testing** (task 8.2 verification, tag → scan → relocate sequence): writing tags (the already-archived add-tag-writing change) physically rewrites the file, changing its size and modification time on disk. But `TagFile.Tag` never updated the tracking store's stored size/mtime after a successful write. The very next scan — including the automatic one that runs at every server startup — compared the file's real (post-tag) stat against the stale (pre-tag) baseline, concluded the file had "changed", and reset its status to `new` and its resolved metadata to blank via the existing changed-file upsert path, while `tagged` stayed `true` (an inconsistent state: tagged, but status `new`). This isn't a narrow edge case — it's the default outcome of tagging anything, ever, since a scan (manual or the automatic startup one) is a completely normal thing to run afterward. It also meant relocation (gated on `identified && Tagged`) would silently stop finding eligible files after any restart following tagging.

Fix, scoped into this change since it directly blocks realistic relocation usage: added `TrackingStore.RecordFileStat(ctx, path, size, modTime int64) error`. `TagFile.Tag` now `os.Stat`s the file immediately after a successful write and calls `RecordFileStat` before `RecordTagged`, refreshing the baseline scan compares against. If the `os.Stat` call itself fails (file vanished right after a successful write — extremely unlikely), `Tag` returns an error rather than silently marking the file tagged with a stale baseline, since that would just recreate the same bug under a different trigger.

## Risks / Trade-offs

- **[Risk] The scan/relocate mutual-exclusion check-then-act window isn't fully atomic** → A `Status()` read and a subsequent `Start()` aren't one atomic operation, so two near-simultaneous triggers (a user clicking Refresh and Relocate within the same instant, from two tabs) could theoretically still both pass their checks. Mitigation: accepted — this narrows an already-narrow race by orders of magnitude (from "a whole scan's duration" to "a few function calls' duration") without adding cross-manager locking machinery to a single-operator tool; not eliminating it in the last few milliseconds is a reasonable trade against that complexity.
- **[Risk] Destination directory sanitization strips characters the user actually wanted** → e.g. an artist name containing a colon becomes a slightly different folder name than expected. Mitigation: this follows the charter's own mandated sanitization list (§4.3) verbatim — not a new design choice, and the details view already shows the resolved (pre-sanitization) metadata for comparison.
- **[Risk] `os.Rename` fails across filesystem/device boundaries (EXDEV)** on some Docker volume configurations (e.g. `/music` bind-mounted from a different filesystem than expected) → Mitigation: since source and destination are both always under the same configured `MUSIC_DIR` root, this is not expected in the documented deployment (`docker-compose.yml`'s single bind mount), but the failure mode if it ever occurs is a clean per-file error (via `os.Rename`'s own error), not a partial move — no silent data loss.
- **[Risk] The best-effort rollback (`Undo`) itself fails after a database write already failed** → Two independent failures in a row (rare, e.g. the disk that just failed a write also can't complete a rename). Mitigation: accepted as a visible, loudly-logged failure rather than something a third automated step tries to fix — the error message states the file's actual current path plainly so an operator can locate and reconcile it by hand; no silent data loss, just an out-of-sync tracking row that clearly says so.
- **[Trade-off] No automatic re-relocation after a later re-identification** → accepted: relocation stays on-demand/human-triggered like every other pipeline step; a corrected identification simply leaves the file where it is with `relocated` cleared until the user re-triggers relocation.

## Migration Plan

- New `relocated`/`relocate_error` columns added via the existing idempotent `ALTER TABLE ... ADD COLUMN` pattern first, same as every prior column addition — this runs before the primary-key rebuild below, so the old table already has the full current column set by the time it's rebuilt.
- **Primary-key rebuild** (new kind of migration, not just an added column): on startup, inspect `PRAGMA table_info(files)`; if the `path` column's `pk` flag is set (the old shape), rebuild the table inside a transaction:
  1. `CREATE TABLE files_new (id INTEGER PRIMARY KEY AUTOINCREMENT, path TEXT NOT NULL UNIQUE, <every other existing column, unchanged>)`
  2. `INSERT INTO files_new (path, format, ...) SELECT path, format, ... FROM files` (explicit column list, not `SELECT *`, so column order differences can't silently misalign values) — `id` autopopulates per row
  3. `DROP TABLE files`, then `ALTER TABLE files_new RENAME TO files`
  4. Recreate the `path` unique index if not implicitly created by the `UNIQUE` column constraint
  
  A database created fresh (no pre-existing `files` table) simply gets the new shape directly from the `CREATE TABLE IF NOT EXISTS` schema constant — the rebuild path only ever runs once, against a database that predates this change, and is a no-op (skipped) on every startup after that.
- No rollback concern for the additive columns. The primary-key rebuild is one-directional (there's no reason to reverse it — the old shape had no advantage); if it's ever needed, restoring from a pre-upgrade database file backup is the rollback path (the same as for any SQLite schema change, no code-level revert).
- Rollback of the relocation *feature* itself: stop calling the new endpoints; already-relocated files remain in their new locations (this is a physical filesystem change, not something a code rollback can undo).
