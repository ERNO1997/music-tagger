## Why

Tagging (just archived) writes resolved metadata, cover art, and lyrics into a file's own tags, but the file still sits wherever it originally was on disk. "Relocate" — physically moving the file into a canonical `Artist/Album/Track - Title` hierarchy — is the last unimplemented step of the core pipeline (project charter §1.3 Phase C step 7 / §4.3), and finishing it completes Scan → Fingerprint → Track → Identify → Enrich → Tag → Relocate end to end.

## What Changes

- Add a `Relocator` usecase port and a `RelocateFile` usecase that, for an already-identified **and tagged** tracked file, sanitizes its resolved artist/album/title (stripping `\ / : * ? " < > |`), zero-pads its track number, builds the destination path `/music/{artist}/{year - }{album}/{track} - {title}.ext` (the release year prefixes the album folder when known, e.g. `2004 - Dead Letters`, and is omitted — not shown as a placeholder — when the release has no usable date), and moves the file there via `os.MkdirAll` then `os.Rename` — sanitizing before both calls, never after (charter §4.3).
- Relocation is atomic per file: on any error (e.g. destination collision, permission failure) the source file is left exactly where it was and the failure is recorded against that file; nothing is partially moved. This extends to the tracking-store write too — if the physical move succeeds but recording the outcome fails, the file is moved back before the call returns, so a failure at either step leaves nothing changed.
- Add an on-demand `RelocateManager` (`POST`/`GET /api/v1/library/relocate`(`/status`)), triggered per-selection from the web UI via a new "Relocate Selected" bulk action — same background-job pattern as identify/enrich/tag, never automatic.
- **Scan and relocate mutually exclude each other**: triggering relocation while a scan refresh is running (or vice versa) is rejected (`409`), rather than allowed to run concurrently like every other pair of job types. A scan walking `/music` while a relocate job moves a file could otherwise briefly see it as both "missing" at its old path and "new" at its new one.
- **Relocation requires the file to be both `identified` and `tagged`**, skipping (and logging, not erroring) any file that's only identified — matching the charter's explicit phase ordering (Enrich → Tag → Relocate) and keeping a file's on-disk tags and its physical location resolved against the same identity at every step, rather than moving a file into a folder named after metadata its own tags don't yet contain.
- **Schema change (internal only, no external API contract exists yet)**: `path` has been the tracking store's primary key and the sole identifier threaded through every port method (`Get`, `RecordIdentification`, `RecordCoverArt`, `RecordTagged`, ...). Relocation is the first capability that changes a file's path after it's tracked, so `path` becomes an ordinary `UNIQUE NOT NULL` column, and a new surrogate `id INTEGER PRIMARY KEY` becomes the row's real, never-changing identity — every existing `TrackingStore` port method keeps its exact signature and `WHERE path = ?` SQL unchanged, since `path` stays uniquely indexed. The one real cost is migration: SQLite can't alter an existing table's declared primary key via `ALTER TABLE`, so a database created before this change needs a one-time full table rebuild on next startup (see design.md).
- Persist a per-file "relocated" outcome (status + optional error), mirroring the existing tagged/tag_error pattern, with re-identification clearing it the same way it already clears tagged/cover art/lyrics — a file's prior location was resolved against a possibly-different prior identity.
- After a successful move, remove the file's original directory and any now-empty ancestor directories, up to (never including) the music root — a directory containing only OS junk files (`.DS_Store`, `Thumbs.db`, `desktop.ini`, AppleDouble `._*` sidecars) counts as empty and has that junk deleted too. Best-effort: a cleanup failure doesn't fail the relocation itself.
- **Found and fixed via live testing, two bugs not originally in scope but directly blocking realistic relocation usage**:
  1. `ErrRefreshInProgress`/`ErrRelocateInProgress`/etc. all alias the same underlying error value, so `errors.Is` could never actually distinguish "blocked by the other job" from "this job's own busy state" — the mutual-exclusion messages were wrong in practice. Fixed with two new, genuinely distinct sentinels (`ErrBlockedByScan`, `ErrBlockedByRelocate`).
  2. Writing tags (the already-archived add-tag-writing change) changes a file's size/mtime on disk, but nothing refreshed the tracking store's stored baseline afterward — so the very next scan (including the automatic one at every server startup) silently reset the file's status to `new` and wiped its resolved metadata, even though it was still correctly tagged. Fixed by having `TagFile.Tag` record the file's post-write size/mtime via a new `TrackingStore.RecordFileStat` method.
  3. The destination-collision check fired on a file's *own* current location — relocating an already-correctly-placed file again (e.g. after a re-identification that resolved back to the same identity) falsely reported "destination already exists" against itself. Fixed by treating a file already at its computed destination as a successful no-op rather than a collision.

## Capabilities

### New Capabilities
- `file-relocation`: On-demand, per-selection physical relocation of an already-identified-and-tagged tracked file into the canonical `Artist/Album/Track - Title` directory hierarchy at its current volume root, with path sanitization and atomic per-file move semantics.

### Modified Capabilities
- `file-tracking-store`: New persisted per-file relocation outcome (relocated/failed + error); `RecordIdentification` additionally clears the relocated outcome (mirroring its existing cover-art/lyrics/tagged reset); the tracked record's `path` becomes mutable — updated by relocation rather than fixed for the record's lifetime.
- `music-library-scan`: `GET /api/v1/library` gains a relocated indicator (and reflects the file's *current* path after relocation); new `POST`/`GET /api/v1/library/relocate` endpoints, mutually exclusive with the existing scan endpoint; web UI gains a "Relocate Selected" bulk action and per-row relocated indicator.

## Impact

- New code: `internal/infrastructure/filestat/relocator.go` (path sanitization + `os.MkdirAll`/`os.Rename`); `internal/usecases/relocate_file.go`, `relocate_manager.go`; a new `Relocator` port and a small `StatusChecker` interface in `internal/usecases/ports.go` (or `job_manager.go`); `internal/infrastructure/web/v1/relocate_handler.go`.
- Schema: new `relocated`/`relocate_error` columns via the existing idempotent migration pattern; a new surrogate `id INTEGER PRIMARY KEY` column with `path` becoming `UNIQUE NOT NULL`, requiring a one-time table-rebuild migration for pre-existing databases (new territory beyond the additive `ALTER TABLE ADD COLUMN` pattern used for every column so far); a new `TrackingStore.RecordRelocation(ctx, oldPath, newPath string) error` method that updates `path` and the relocated outcome in one statement (SQL unchanged by the schema change).
- Two new API endpoints (`POST`/`GET /api/v1/library/relocate`), same shape as the existing tag trigger/status pair; no changes to scan/identify/enrich/tag endpoints' request/response *shapes* — but `POST /api/v1/library/scan` and `POST /api/v1/library/relocate` can now each reject the other's concurrent request (both still `409`, message distinguishes which job is blocking).
- Filesystem: writes to `/music` for the second time (after tagging) — but now also renaming/moving files and creating directories, a materially larger blast radius than an in-place tag write. Every write path must be exercised against real file copies before running against the user's real library, per the tagging change's precedent.
- `RefreshManager` gains one new method (`SetRelocateStatus`) called from the composition root after `RelocateManager` is constructed — a small, backward-compatible addition, not a signature change to its existing constructor or `Start`/`Status` methods.
